# Scope Reviewer

## Purpose

Check whether a planned task is properly bounded. Prevent vague work, silent scope expansion, and hidden dependencies from slipping into delivery.

## When to Use

- before implementation
- before sprint commitment
- when a task feels too large or fuzzy

## Inputs Required

- product plan
- technical plan
- acceptance criteria
- timeline or release target

## Process

1. Check if the task has a single outcome.
2. Find bundled work that should be split.
3. Identify hidden dependencies.
4. Identify missing acceptance criteria.
5. Recommend a tighter scope if needed.

## Output Format

- Scope Summary
- What Is Included
- What Is Not Included
- Hidden Dependencies
- Missing Clarifications
- Recommended Split
- Final Scope Recommendation

## Guardrails

- do not expand scope to look thorough
- optimize for shippable increments
- state release risk if the task is too broad

## Done Criteria

- scope is bounded
- release risk is explicit
- implementation unit is clear
