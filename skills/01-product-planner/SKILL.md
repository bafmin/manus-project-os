# Product Planner

## Purpose

Pressure-test whether the requested feature solves the right problem. Clarify user outcome, business value, workflow impact, and what should be in or out of scope before build work starts.

## When to Use

- before building a new feature
- when a request sounds too narrow or implementation-led
- when the team needs stronger product framing

## Inputs Required

- feature request
- project charter
- product requirements
- known user role
- business objective

## Process

1. Rewrite the request in user-outcome terms.
2. Identify the real user job.
3. Identify what success looks like.
4. Challenge the narrow interpretation if needed.
5. Separate MVP from later enhancements.
6. List edge cases and risks.

## Output Format

- Feature Request Summary
- Real User Problem
- Business Goal
- Recommended Scope
- Out of Scope
- Acceptance Criteria Draft
- Product Risks
- MVP Recommendation

## Guardrails

- do not jump into architecture
- do not recommend hidden scope creep without stating cost
- keep the recommendation tied to user and business value

## Done Criteria

- problem is clearly framed
- success is measurable
- scope is clear enough for technical planning
