# FEATURE_REQUEST_TEMPLATE

A feature request defines a proposed product capability before implementation planning begins.

This document should be completed before a change plan or task brief is created.

---

## Feature Name

Provide a clear, descriptive name for the feature.

---

## Summary

Describe the feature in plain language.

State what the feature does and why it matters.

---

## Problem Being Solved

Explain the operational or user problem this feature addresses.

Include the current pain point, limitation, or business issue.

---

## Target Users

Identify who will use or benefit from this feature.

Examples

* administrators
* operators
* internal staff
* customers
* partners

---

## Business Value

Explain why this feature is worth building.

Examples

* reduces manual work
* improves onboarding
* increases accuracy
* supports growth
* reduces support requests

---

## User Story

Format:

As a [type of user], I want to [action], so that [outcome].

Example

As an administrator, I want to invite users by email, so that I can onboard team members without manual account creation.

---

## Current State

Describe how the system works today.

Include relevant limitations or workarounds.

---

## Desired State

Describe what should be true after the feature is implemented.

---

## Affected Areas

* frontend
* backend
* database
* authentication

* reporting

## Risks and Concerns

List potential issues or questions.

* risk
* risk

Examples

* email delivery reliability
* permission edge cases
* migration complexity

---

## Security Considerations

Document security or permissions concerns.

Examples

* role based access

* input validation
* rate limiting

---

## Success Metrics

Define how success will be measured.

Examples

* adoption rate

* fewer support tickets
* faster onboarding
